## Rationale

`RabbitMqConnectionProvider` intentionally caches one completed connection task for the outbound topology lifetime. For the first release, USF should make the corresponding recovery strategy explicit: RabbitMQ.Client automatic recovery owns reconnect attempts and restores the cached connection wrapper after transient network failures. USF should require that behavior instead of adding provider-level invalidation on `ConnectionShutdownAsync`, which would compete with the client's recovery loop.

RabbitMQ.Client also restores autorecovering channel wrappers by replacing their inner channels and then raising `IRecoverable.RecoveryAsync`. The channel pool already refuses to lease a channel after observing `ChannelShutdownAsync`, but it treats that observation as permanent. It must recognize the recovery event so an idle pooled wrapper that the client successfully restores becomes leasable again. Automatic recovery is an availability mechanism, not a delivery guarantee: it does not buffer or republish messages, and callers still need retry or an outbox for publishes that fail or time out during an outage.

## Acceptance Criteria

- [ ] A RabbitMQ connection factory with `AutomaticRecoveryEnabled == false` is rejected with a clear `OutboundTopologyValidationException` before USF attempts to create a connection. USF does not silently override the caller's factory configuration.
- [ ] `RabbitMqConnectionProvider` continues to cache one successfully created `IConnection` for the topology lifetime and does not invalidate it on `ConnectionShutdownAsync`; RabbitMQ.Client owns transient reconnection.
- [ ] A pooled channel that observes `ChannelShutdownAsync` remains unavailable while closed, but becomes healthy and reusable when the same autorecovering wrapper raises `IRecoverable.RecoveryAsync` and reports `IsOpen == true`.
- [ ] A channel that closes without recovering is still discarded, decrements the pool's live-channel count exactly once, and can be replaced lazily without leaking a lease slot.
- [ ] Pool disposal unsubscribes both shutdown and recovery event handlers before disposing a channel.
- [ ] `TopologyRecoveryEnabled` remains caller-controlled. XML documentation states that RabbitMQ.Client defaults it to `true`, explains that it controls exchange, queue, and binding recovery rather than connection recovery, and does not require it when broker topology is managed externally.
- [ ] USF logs connection shutdown, successful recovery, and failed recovery attempts with structured fields while leaving RabbitMQ.Client in control of reconnection. These notifications do not invalidate the cached connection.
- [ ] XML documentation for RabbitMQ connection-factory configuration states that automatic recovery is required and that `NetworkRecoveryInterval` and `TopologyRecoveryEnabled` remain caller-configurable.
- [ ] A RabbitMQ transport README documents that automatic recovery does not buffer or republish failed or in-flight messages, so applications requiring at-least-once effects must retry safely or use an outbox.
- [ ] The RabbitMQ transport README states that RabbitMQ cluster nodes should use a consistent `channel_max`: USF validates the channel budget on the initial connection, while RabbitMQ.Client performs subsequent recovery internally.
- [ ] Automated tests are written, including: rejection of a factory with automatic recovery disabled; reuse of the same pooled channel wrapper after shutdown followed by recovery; discard and replacement when shutdown is not followed by recovery; no live-channel-count or lease-slot leak; structured lifecycle logs; and installation and disposal of connection-event subscriptions exactly once.
- [ ] An integration test restarts the RabbitMQ broker after a channel has entered a maximum-size-1 pool, verifies that a publish attempted during the outage fails or times out within a finite bound, waits for RabbitMQ.Client recovery, and verifies that repeated subsequent confirmed publishes succeed without pool exhaustion.

## Technical Details

Keep reconnection ownership inside RabbitMQ.Client. In `RabbitMqOutboundTopologyCompiler.CreateConnectionAsync`, validate the `ConnectionFactory` returned by the configured delegate immediately before calling `CreateConnectionAsync`. If `AutomaticRecoveryEnabled` is false, throw an `OutboundTopologyValidationException` with an actionable message. Do not mutate the factory and do not subscribe `RabbitMqConnectionProvider` to `ConnectionShutdownAsync`: RabbitMQ.Client returns an autorecovering `IConnection` wrapper when automatic recovery is enabled, and the provider should keep handing out that stable wrapper.

Add structured recovery observability when USF first obtains the connection. Subscribe to `IConnection.ConnectionShutdownAsync`, `RecoverySucceededAsync`, and `ConnectionRecoveryErrorAsync`, logging the connection lifecycle transition and the relevant shutdown reason or recovery exception. These event handlers are diagnostic only: they must not clear the provider cache, create a replacement connection, or start a competing recovery loop. Ensure subscriptions are installed once for the cached connection lifecycle and removed during topology disposal.

Extend `DefaultRabbitMqChannelPool.PooledChannel` to recognize recoverable channels. When the supplied `IChannel` implements `IRecoverable`, subscribe to `RecoveryAsync` in addition to `ChannelShutdownAsync`. The shutdown handler continues to set `_observedShutdown`; the recovery handler clears it with `Interlocked.Exchange` only after RabbitMQ.Client reports recovery. `IsHealthy` must continue to require both a clear observed-shutdown flag and `IChannel.IsOpen`, so the pool never leases the wrapper during the outage. Unsubscribe both handlers during disposal. A wrapper discarded because a caller acquires during the outage remains discarded; a later acquisition can create a fresh channel after the cached autorecovering connection has recovered.

Preserve `TopologyRecoveryEnabled` as a caller decision rather than a USF requirement. It defaults to `true` in RabbitMQ.Client and is useful when the client should recreate declared exchanges, queues, and bindings, but deployments that provision durable topology externally may disable it while still relying on connection recovery. Add XML comments on both `RabbitMqOutboundTopologyBuilder.UseConnectionFactory(...)` overloads to document the required and caller-controlled settings. Add `src/Transports/Usf.Transport.RabbitMq/README.md` for the operational guidance: publish behavior during an outage, retry or outbox responsibility, and the requirement that clustered broker nodes use a consistent `channel_max`.

Unit-test recovery behavior with a hand-crafted recoverable `IChannel` test double that can raise shutdown and recovery events and transition `IsOpen`. Unit tests should prove reuse of the same pooled wrapper after recovery and verify that lifecycle event subscriptions are installed only once and removed during disposal. For the integration test, configure a short `NetworkRecoveryInterval`, use an explicit maximum-size-1 channel group with confirms enabled, publish once to populate the pool, restart the Testcontainers RabbitMQ broker, and attempt a publish during the outage with a finite timeout. Wait for the client's recovery signal, then perform repeated confirmed publishes. The integration test should prove externally observable availability and bounded failure behavior rather than the identity of RabbitMQ.Client's internal autorecovering wrapper.
